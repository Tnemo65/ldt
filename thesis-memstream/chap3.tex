\section{Proposed Method}

\subsection{Overview}

CA-DQStream implements a four-layer processing pipeline on Apache Flink that separates infrastructure concerns from algorithmic contributions. The pipeline architecture (Figure~\ref{fig:system_architecture}) flows as follows: Layer 1 (Schema Filter and Deduplication) validates completeness and removes duplicates via Flink KeyedState with 7-day TTL. Layer 2 (Rendezvous Architecture) forks into parallel Canary and Complex branches that join at the MetaAggregator. Layer 3 (MetaAggregator) computes 1-minute windowed quality metrics. Layer 4 (IEC) monitors meta-streams via ADWIN and triggers multi-strategy self-adaptation.

\subsection{Feature Engineering}

The Online Autoencoder operates on a 21-dimensional enhanced feature vector comprising 15 base features and 6 ratio features. Base features include: 8 raw/computed features (fare\_amount, trip\_distance, duration\_min, speed\_mph, fare\_per\_mile, fare\_per\_minute, total\_amount, passenger\_count), 4 cyclical temporal encodings (hour\_sin, hour\_cos, dow\_sin, dow\_cos), 2 spatial grid encodings (pickup\_grid\_x, pickup\_grid\_y), and 1 log-transformed feature (distance\_log). Ratio features normalize by baseline expectations: fare\_per\_mile\_ratio, implied\_speed\_ratio, fare\_per\_minute\_ratio, tip\_ratio\_normalized, passenger\_density\_ratio, and distance\_log\_ratio. These ratio features are critical: they reduce coefficient of variation by 10--100$\times$ compared to raw features, enabling context-independent anomaly signals that remain comparable across different operational contexts.

Following established practices from NYC TLC taxi feature engineering \citep{nyctlc}, the system computes Haversine distance from pickup/dropoff coordinates, total travel time in minutes, and meter increment patterns (the per-unit billing increment that reflects fare regulation). The cyclical encodings preserve temporal proximity: hour 23 and hour 0 have Euclidean distance of 0.26 on the unit circle, matching the distance between adjacent hours, which is critical for midnight transition anomalies. The spatial grid maps 265 TLC zones to a 4$\times$4 coordinate system (16 cells), preserving geographic locality while ensuring sufficient sample density per cell for reliable K-Means clustering. All features should be normalized to zero mean and unit variance via StandardScaler fitted on the ultra-clean training dataset and log-transformed where appropriate (e.g., $\log(1 + \text{trip\_distance})$), eliminating seasonal artifacts where identical fares produce different anomaly scores at different times of year \citep{sklearn_api}.

\subsection{Layer 1: Baseline Validation}

Layer 1 addresses the Completeness dimension of DAMA-DMBOK through two sub-stages: schema validation and deduplication. Schema validation performs deterministic NULL checks and range validation on all incoming records, leveraging Confluent's Schema Registry for enforcement of data contracts at the entry point of the streaming pipeline \citep{confluent_schema_registry}. Beyond structural schema enforcement, CA-DQStream employs CEL-based (Common Expression Language) data contracts for semantic validation of value ranges, patterns, and cross-field dependencies \citep{confluent_dq_patterns}. This distinction mirrors Grab's approach: syntactic validation (schema conformance) catches structural violations, while semantic validation (value ranges, cross-field consistency) catches logically or semantically invalid records \citep{grab_streaming_dq}. For instance, a record passing schema validation may still fail semantic checks if fare\_amount is negative or if speed exceeds physical limits. Schema evolution is managed through compatibility modes: BACKWARD (new schema accepts old data), FORWARD (old schema accepts new data), and FULL (bidirectional compatibility) \citep{confluent_schema_registry}. Grab's Coban platform defines data quality tests using FlinkSQL-based specifications, enabling declarative rule definitions that are both human-readable and computationally efficient \citep{grab_coban_platform}.

Schema validation performs deterministic NULL checks and range validation on all incoming records:
\begin{itemize}
    \item \texttt{passenger\_count IS NOT NULL} and \texttt{passenger\_count $\in$ [1, 9]}
    \item \texttt{RatecodeID IS NOT NULL} and \texttt{RatecodeID $\in$ \{1, 2, 3, 4, 5, 6, 99\}}
    \item \texttt{VendorID IS NOT NULL} and \texttt{VendorID $\in$ \{1, 2, 6\}}
    \item \texttt{pickup\_datetime IS NOT NULL}, \texttt{dropoff\_datetime IS NOT NULL}
\end{itemize}

Deduplication uses Flink KeyedState with a composite key: (tpep\_pickup\_datetime, PULocationID, fare\_amount, passenger\_count), stored with 7-day TTL via RocksDB backend. Records failing schema validation are emitted to the \texttt{dq-schema-violations} Kafka topic; records passing schema validation proceed to Layer 2.

Scale: Approximately 310,000 records out of 3M (10.1\%) are rejected at Schema Validation, with the dominant rejection being NULL passenger\_count/RatecodeID (9.1\% of records affected by the same upstream data capture issue).

\subsection{Layer 2: Memory-Based Streaming Anomaly Detection}

Layer 2 implements the Rendezvous architecture, forking the data stream after Schema Validation into two parallel branches that join at the MetaAggregator. The ML-based anomaly detection draws on MemStream (WWW'22), which combines a Denoising Autoencoder with a Memory Module and FIFO buffer to detect anomalies in data streams without requiring labeled data \citep{bhatia2022memstream}. The Memory Module stores $K=5$ nearest neighbor encoded normal patterns with an exponentially weighted discounting scheme to prioritize recent memories. The scoring mechanism computes the anomaly score as:

\begin{equation}
\text{Score}(\mathbf{z}_t) = \sum_{i=1}^{K} \gamma^{i-1} \, R(\mathbf{z}_t, \hat{\mathbf{z}}_i^t)
\label{eq:memstream_score}
\end{equation}

where $R$ denotes the L1 distance between the encoded incoming observation $\mathbf{z}_t$ and the $i$-th nearest neighbor $\hat{\mathbf{z}}_i^t$ from memory, $\gamma=0.5$ is the discount factor that exponentially down-weights older memories, and $K=5$ nearest neighbors are considered \citep{bhatia2022memstream}. The Memory Module update threshold is set to $\beta=0.01$, meaning only records with anomaly score below this threshold are used to update memory, preventing memory poisoning from anomalous data \citep{bhatia2022memstream}. Theoretical analysis from MemStream establishes that required memory size is proportional to the ratio of distribution spread to drift speed, providing principled guidance for memory allocation in non-stationary environments. ADWIN is used for concept drift detection within the memory module, enabling the system to recognize when stored patterns no longer reflect current data distribution \citep{bifet2007learning}.

\textbf{Canary Branch (Rule-Based):} Six deterministic business rules evaluate validity and consistency constraints:
\begin{itemize}
    \item \texttt{fare\_amount $\geq$ 0} (negative fare: voided trips, data entry errors)
    \item \texttt{trip\_distance $>$ 0} (zero distance: tolls-only, canceled trips)
    \item \texttt{dropoff\_datetime $>$ pickup\_datetime} (impossible timestamp: system errors)
    \item \texttt{speed\_mph $<$ 100} (GPS/meter malfunction)
    \item \texttt{fare\_amount $\leq$ 500} (unreasonably high: meter tampering)
    \item \texttt{payment\_type$\neq$1 $\to$ tip\_amount = 0} (payment inconsistency)
\end{itemize}

The Canary branch processes each record in under 5ms, acting as a lightweight sanity check. Records flagged by Canary bypass the Complex branch entirely, reducing average processing time by 6.75ms per record.

\textbf{Complex Branch (ML-Based):} The Online Autoencoder with Memory Module scores multivariate anomalies. The model architecture is: 25D input $\rightarrow$ 50D hidden $\rightarrow$ 25D output (Denoising Autoencoder with Tanh activation, Gaussian noise injection std=0.1). The Memory Module stores 1,000 encoded normal patterns in FIFO buffer, scoring via K-nearest neighbors (K=5) with exponentially weighted discounting.

Scoring mechanism: For each incoming record $\mathbf{x}$: (1) encode: $\mathbf{z} = \text{encoder}(\mathbf{x})$; (2) reconstruct: $\mathbf{\hat{x}} = \text{decoder}(\mathbf{z})$; (3) reconstruction error: $R(\mathbf{x}, \mathbf{\hat{x}}) = \|\mathbf{x} - \mathbf{\hat{x}}\|_2^2$; (4) memory distance: mean L1 distance to K nearest neighbors; (5) final score: $\text{Score}(\mathbf{z}) = \max(\text{reconstruction\_error}, \text{memory\_distance})$.

\textbf{ContextBeta Thresholding:} The anomaly score is compared against the 4D context-aware threshold matrix. Each cell in the 4D matrix (trip\_type $\times$ time\_window $\times$ day\_type $\times$ neighborhood\_id, 588 cells total) stores $T_{\text{cell}} = \mu_{\text{cell}} + 2\sigma_{\text{cell}}$, computed from the ultra-clean training dataset. Cells with fewer than 100 samples fall back to the global threshold $T_{\text{global}} = \mu_{\text{all}} + 2\sigma_{\text{all}} = \$45.46$ to avoid overfitting to noise.

\textbf{Fault Tolerance:} The Rendezvous architecture provides graceful degradation: if the Complex branch is unavailable, Canary continues processing uninterrupted. The MetaAggregator synchronizes the two branches via MapState with 5-second TTL, handling out-of-order arrivals.

PyFlink Vectorized UDFs process records in mini-batches (batch\_size=500), achieving 16$\times$ throughput improvement over scalar Python UDFs by amortizing JVM-Python IPC overhead and enabling NumPy's BLAS-accelerated operations.

\subsection{Layer 3: Meta-Aggregation}

The MetaAggregator serves two functions: data routing and meta-stream generation. After Canary and Complex complete their evaluations, results converge at the MetaAggregator, which emits clean records and violation records to appropriate Kafka topics and MinIO buckets. ADWIN-U (KAIS'25) provides the reference implementation for the meta-aggregation layer, comparing a reference window against the current window using unsupervised divergence metrics \citep{adwin2025}. Crucially, higher-order statistics such as Skewness and Kurtosis outperform traditional mean and variance for unsupervised drift detection, as they capture distributional shape changes that first-order statistics miss \citep{adwin2025}. The BAR metric (Balanced Accuracy by Requested Labeled Data) is used when labeled feedback is available to calibrate the meta-aggregation thresholds \citep{adwin2025}. 

Confluent's data quality feedback loop provides the operational model for the MetaAggregator: validation results feed into a quarantine mechanism, monitored via dashboards with threshold alerting, triggering corrective action when quality degrades \citep{confluent_dq_feedback_loop}. The five KPIs tracked by CA-DQStream's meta-aggregation align with industry practice: completeness (proportion of valid records), accuracy (proximity to true values), freshness (event arrival lag vs. processing lag), error rates (violation frequency), and quarantine volume (records held for review) \citep{confluent_dq_feedback_loop}.

The quality meta-stream consists of six metrics computed over 1-minute windows grouped by neighborhood:

\begin{align}
\text{null\_rate}(t) &= \frac{|\{\,\text{NULL records at }t\,\}|}{|\{\,\text{total records at }t\,\}|} \label{eq:null_rate} \\[6pt]
\text{violation\_rate}(t) &= \frac{|\{\,\text{BR violations at }t\,\}|}{|\{\,\text{total records at }t\,\}|} \label{eq:violation_rate} \\[6pt]
\text{anomaly\_rate}(t) &= \frac{|\{\,\text{AE anomalies at }t\,\}|}{|\{\,\text{total records at }t\,\}|} \label{eq:anomaly_rate} \\[6pt]
\text{volume}(t) &= |\{\,\text{records processed at }t\,\}| \quad \text{per minute} \label{eq:volume} \\[6pt]
\text{dedup\_rate}(t) &= \frac{|\{\,\text{duplicates detected at }t\,\}|}{|\{\,\text{total at }t\,\}|} \label{eq:dedup_rate} \\[6pt]
\Delta\_\text{score}(t) &= \frac{\big|\,\text{violation\_rate}(t) - \text{anomaly\_rate}(t)\,\big|}
                               {\text{violation\_rate}(t) + \text{anomaly\_rate}(t) + \epsilon} \label{eq:delta_score}
\end{align}

The $\Delta$\_score (Canary-Complex Divergence) measures normalized disagreement between rule-based and ML-based anomaly classifications. When $\Delta$\_score increases, the ML model's learned patterns no longer align with current data distribution, signaling concept uncertainty that ADWIN monitors.

\subsection{Layer 4: Intelligent Evolution Controller}

The Intelligent Evolution Controller (IEC) monitors quality meta-streams via ADWIN and triggers multi-strategy self-adaptation when drift is detected. The IEC classifies drift into five scenarios based on meta-stream trends. METER (MEta-learning-based adapTER, WWW'23) \citep{meter_2023} uses an MLP hypernetwork (64-32-16 architecture) trained on historical drift patterns to predict optimal K-Means centroid displacements from current meta-metrics, providing the algorithmic foundation for moderate drift adaptation where cluster boundaries shift but overall structure remains. DTD (Dynamic Threshold Discovery, AAAI'26) \citep{dtd_2026} formally proves that no single fixed threshold is universally optimal, and introduces a 3-way comparison framework: (1) old model retains too late (drift is missed), (2) new model applies at correct timing (optimal adaptation), (3) false alarm (no drift present, switching would be harmful). The SCAR benchmark (Streaming Concept Adaptation with Reconstruction, 2025) \citep{scar_2025} establishes that periodic retraining (reconstruction strategy) is a surprisingly strong baseline, validating CA-DQStream's continuous evolution approach. ADWIN (Adaptive Windowing, SIAM'07) \citep{bifet2007learning} provides the core drift detection mechanism: when the mean difference between a reference window and a current window exceeds a statistically computed threshold $\varepsilon$, drift is declared. The IEC monitors three key signals for adaptation decisions: Importance (drift magnitude), Error (detection rate degradation), and Correctness (false positive rate increase), coordinating these signals to select the optimal adaptation strategy.

Four adaptation strategies are available, applied in sequential fallback order:

\begin{enumerate}
    \item \textbf{Continuous Evolution:} Incrementally update K-Means centroids via \texttt{mini-batch K-Means partial\_fit()} on recent data windows; Memory Module continues FIFO updates on clean data. Used for gradual drift with slow metric changes (e.g., seasonal fare increase). Cost: seconds, no downtime.

    \item \textbf{Switching Scheme:} Maintain pre-trained model ensemble (weekday model, weekend model, seasonal models) and switch based on temporal context. Used for recurring drift with cyclic patterns (e.g., weekend vs. weekday traffic). Cost: negligible (model swap via Broadcast State).

    \item \textbf{METER Adaptation:} METER (MEta-learning-based adapTER) uses an MLP hypernetwork (64-32-16 architecture) trained on historical drift patterns to predict optimal K-Means centroid displacements from current meta-metrics. Used for moderate drift where cluster boundaries shift but overall structure remains. Cost: 2--5 minutes, adaptation latency while current model continues serving traffic.

    \item \textbf{Spatial Tracking:} Train per-neighborhood models for affected zones, keep global model for others. Used for localized drift affecting specific geographic areas (e.g., construction closes major route). Cost: hours for multi-model training. Rare: only 1\% of drift events.
\end{enumerate}

The IEC executes a 3-tier self-adaptation protocol. Tier 1 (within 1 minute): emit drift alert to Prometheus/Grafana, expand monitoring window. Tier 2 (within 10 minutes): recalculate relevant thresholds via A/B validation, commit if new FPR improves. Tier 3 (within 1 hour): trigger model update (memory reset, METER shift, or full retrain), publish to \texttt{if-model-updates} Kafka topic, deliver to all Flink instances via Broadcast State.

Zero-downtime model swap: the Flink pipeline continues processing records during model updates. The current model serves traffic while the new model downloads asynchronously. An atomic volatile reference swap provides sub-millisecond swap latency with no pipeline interruption.

On model update failures, an Action Replay Worker retries with exponential backoff ($2^n$ seconds, up to 10 retries), with a 15-minute cooldown lock preventing retry storms during sustained outages.

\subsection{Fault Tolerance}

CA-DQStream achieves end-to-end exactly-once semantics through two complementary mechanisms. First, Flink Checkpointing implements Chandy-Lamport distributed snapshots \citep{chandy_lamport_1985}, capturing periodic state snapshots (45-second intervals) to MinIO S3 storage, enabling failure recovery without data loss. Second, Kafka Transactions use transactional writes with \texttt{transaction.timeout.ms = 10 minutes}; on checkpoint recovery, in-flight transactions are aborted and replayed idempotently \citep{confluent_dq_patterns}. The dead-letter queue (DLQ) pattern from Confluent \citep{confluent_dead_letter_queue} handles records that fail validation: invalid records are diverted to a quarantine topic rather than blocking the pipeline. However, practitioners should note that DLQ creates error-free but \textit{incomplete} streams---using DLQ without monitoring leads to silent data loss, where records are removed from the stream but never corrected or reprocessed \citep{confluent_dead_letter_queue}. For atomicity across Kafka and external databases, CA-DQStream employs two-phase commit (2PC) coordination, ensuring that either all operations in a transaction commit or none do. Idempotent operations are used throughout via upsert modes with unique keys, allowing safe reprocessing of records after failures without creating duplicates \citep{confluent_dq_patterns}.

State size is carefully managed: the deduplication operator stores Boolean flags (not full records) per trip\_id within the 7-day TTL window, approximately 56 MB per slot with RocksDB overhead. Bloom filters reduce RocksDB I/O by 90\%. The MetaAggregator uses the AggregateFunction pattern to prevent OOM on large windows.

Model distribution: PyFlink executes Python UDFs in separate worker processes per TaskManager slot. Broadcast State distributes model parameters and memory state across these workers for in-process scoring with microsecond access latency. A 4-node cluster generates approximately 768 MB aggregate network traffic per model update, with approximately 150--200ms GC pause on each receiving TaskManager (G1GC, 4GB heap). With update frequency of 2--3 times/day, total GC overhead is less than 0.01\% of daily uptime.

The pipeline uses 7 Kafka topics with 12 partitions each: \texttt{taxi-nyc-raw} (raw NYC TLC records), \texttt{dq-schema-violations}, \texttt{dq-hard-rule-violations}, \texttt{dq-if-anomalies}, \texttt{dq-anomaly-scores}, \texttt{dq-meta-stream}, and \texttt{if-model-updates}.

MinIO stores all pipeline outputs via Flink StreamingFileSink in Parquet format (columnar compression provides 3--5$\times$ storage reduction over row-based formats): \texttt{cadqstream-raw}, \texttt{cadqstream-violations}, \texttt{cadqstream-anomalies}, \texttt{cadqstream-metrics}, \texttt{cadqstream-drift}, \texttt{cadqstream-checkpoints}, and \texttt{mlflow-artifacts}.

\section{Chapter Summary}

This chapter presented the CA-DQStream framework, a context-aware streaming data quality monitoring system built on four core architectural innovations. The 4D Context-Aware Thresholding framework partitions the data space into 588 context cells, reducing false positive rates compared to global thresholds, with particular benefit for high-value data segments where global thresholds catastrophically fail. The Rendezvous Pipeline Architecture parallelizes rule-based and ML-based anomaly detection through fork-join execution, improving throughput and reducing average latency compared to sequential pipelines. The Intelligent Evolution Controller with four adaptation strategies maintains model accuracy under concept drift, preferring lightweight incremental updates to full retraining for computational efficiency. The hybrid K-Means plus Online Autoencoder with Memory Module detects multivariate anomalies that business rules cannot capture, with continuous memory-based learning that adapts to evolving distributions.

The experimental evaluation in Chapter 5 validates these contributions through comprehensive ablation studies and comparative baselines, demonstrating that CA-DQStream achieves production-grade data quality monitoring with context awareness, low latency, high throughput, and autonomous adaptation to concept drift.
