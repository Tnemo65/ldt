2.4. NAB Events (Numenta Anomaly Benchmark — Reference Patterns)

  Nguồn: https://github.com/numenta/NAB
  Paper: https://arxiv.org/abs/1510.03336

  NAB là synthetic dataset để benchmark anomaly detection methods.
  Cung cấp ground truth labels + profile scoring (α=0.1, β=1/144).

  ┌──────────────────────────────────────────────────────────────────────────────┐
  │ PHÂN BIỆT: NAB Events vs Concept Drift                                       │
  ├──────────────────────────────────────────────────────────────────────────────┤
  │ NAB Events    → POINT anomaly patterns (reference). Dùng để benchmark       │
  │                 model detect spike/drop spikes. MAP TỚI Point Anomaly        │
  │                 (Section 2.5 / BƯỚC 5).                                     │
  │                 → 5 sự kiện bên dưới: sudden drop, massive drop, gap,     │
  │                   spike, extreme — đều là POINT anomalies.                 │
  │                                                                              │
  │ Concept Drift → Phân bố data thay đổi từ từ (không phải spike/drop).      │
  │                 KHÔNG liên quan NAB. MAP TỚI 3 drift polluters bên dưới.   │
  │                 → Abrupt, Incremental, Cyclical.                             │
  └────────────────────────────────────────────────────────────────────────────┘

  5 sự kiện tham khảo (2014-2015) — POINT ANOMALY PATTERNS:
  - NYC Marathon:   02/11/2014  (sudden drop)
  - Thanksgiving:   27/11/2014  (massive drop)
  - Christmas:      25/12/2014  (gap day)
  - New Year:       01/01/2015  (spike/drop)
  - Blizzard:       26-27/01/2015 (extreme)

  Visualization: BƯỚC 6 (EDA on INJECTED) visualize theo ground_truth_mask.npy:
  - fig/injected_drift_abrupt_timeseries.png    → Abrupt drift (point injection)
  - fig/injected_drift_incremental_rolling.png  → Incremental drift
  - fig/injected_drift_cyclical_acf.png         → Cyclical drift (4-week ACF peak)
  - fig/injected_ground_truth_coverage_bar.png → Point anomaly rate by type
  - fig/injected_label_stacked_bar.png          → % per polluter type (stacked)

  NOTE: NAB scoring profile (α, β) không áp dụng trực tiếp vì production
  data không có NAB labels. Chỉ dùng NAB events làm QUALITATIVE REFERENCE.


2.5. Icewafl Framework (EDBT 2025) — Data Polluter Architecture

  Icewafl (Inserting Customizable Errors with Apache Flink) là framework
  được công bố tại EDBT 2025 (Barcelona) bởi Schinninger et al.
  GitHub: https://github.com/chri-schi/Icewafl

  Framework này xác lập polluter architecture chuẩn:

    Polluter = (e, c, A_p)
      e  = event (thời điểm/khoảng thời gian)
      c  = condition (điều kiện kích hoạt)
      A_p = attribute perturbation (cách thay đổi giá trị)

  Kiến trúc Icewafl gồm 4 thành phần chính:
  1. Builder      : Xây dựng pollution pipeline (PolluterBuilder pattern)
  2. Conditions   : Điều kiện kích hoạt (temporal, probability, function, composite)
  3. Polluters    : Attribute perturbation (replace, multiply, reduce precision, composite)
  4. Read/Write   : Data I/O (CSV, Parquet, streaming)

  Áp dụng vào project:
  - Sử dụng Icewafl-inspired polluter thay vì hardcoded inject
  - Từng polluter có: type (point/concept), condition, rate, attributes affected
  - Ground truth mask per polluter type (để phân tách point vs concept drift detection)

BƯỚC 3: CLEAN — Tạo Clean Dataset
---------------------------------
- Script: scripts/clean_taxi_data.py
- Pipeline 3 tầng: Schema → Hard Rule → ML Detection

  ═══════════════════════════════════════════════════════
  TẦNG 1: SCHEMA CLEANING
  ═══════════════════════════════════════════════════════
  - Validate columns: drop unknown columns, rename TLC fields
  - Parse datetime: tpep_pickup_datetime, tpep_dropoff_datetime
  - Cast types: float64 for numerics, int for IDs
  - Handle encoding: unicode, trailing whitespace
  - Deduplicate column names (duplicate column names from parquet)
  - Output: schema-validated records

  ═══════════════════════════════════════════════════════
  TẦNG 2: HARD RULE CLEANING (Rule-Based)
  ═══════════════════════════════════════════════════════
  - Loại bỏ semantic violations tuyệt đối

  Cleaning Rules:
  - fare_amount >= 0
  - trip_distance >= 0
  - passenger_count: 1-6
  - RatecodeID: [1, 2, 3, 4, 5, 6, 99] only
  - PULocationID / DOLocationID: 1-263
  - duration_s: > 0 và < 86400
  - speed_mph: > 0 và <= 80

  Output: data/clean/tier1_hardrule.parquet

  ═══════════════════════════════════════════════════════
  TẦNG 3: ML-BASED ANOMALY DETECTION (Residual Cleaning)
  ═══════════════════════════════════════════════════════
  - Mục tiêu: Phát hiện anomalies còn sót lại sau hard rule
  - Method: Dùng batch ML models (KHÔNG dùng MemStream để
    tránh circular dependency)

  Models:
  ┌────────────────┬────────────────────────────────────────────────────────┐
  │ Model          │ Config                                                  │
  ├────────────────┼────────────────────────────────────────────────────────┤
  │ IsolationForest│ n_estimators=200, max_samples=512, contamination=0.01 │
  │ HBOS           │ n_bins=20, alpha=0.05, contamination=0.01             │
  │ Stat-threshold │ fare_per_mile > p99 OR speed > p99 theo context cell │
  └────────────────┴────────────────────────────────────────────────────────┘

  Approach:
  1. Fit IF + HBOS trên tier1 data (all months combined)
  2. Ensemble: record là anomaly nếu ≥ 2/3 models flag
  3. Human review sample (top 100 scored records) trước khi drop
  4. Soft flag: ghi log anomaly potential để có thể revert

  NOTE: quantile-based threshold per context cell (10 NB × 8 time contexts)
  NOTE: Không dùng MemStream vì circular — MemStream cần clean data
        để train, nên không thể dùng chính nó để clean

  Output:
  - data/clean/clean_2024.parquet       (Jan 2024 - Dec 2024, 0% anomaly)
  - data/clean/clean_2025.parquet       (Jan 2025 - Dec 2025, 0% anomaly)
  - data/clean/clean_2024_log.json      (Chi tiết: records removed, by which tier/rule)
  - data/clean/clean_2025_log.json
  - data/clean/ml_flagged.parquet       (Records flagged by ML, for review)

BƯỚC 5: INJECT — Icewafl-Inspired Polluter (Data Polluter)
----------------------------------------------------------
- Script: scripts/inject_polluter.py
- Framework: p = (e, c, A_p) — theo Icewafl EDBT 2025

  Polluter Pipeline (Icewafl-inspired):
  ┌────────────────────────────────────────────────────────────────────────────────────┐
  │ Polluter             │ Type    │ Rate  │ Condition    │ Attributes              │
  ├──────────────────────┼─────────┼───────┼──────────────┼────────────────────────┤
  │ NULLLocationPolluter │ POINT   │ 2.0%  │ always       │ PULocationID,DOLocID   │
  │ PlaceholderPolluter  │ POINT   │ 0.1%  │ always       │ passenger_count        │
  │ NegativeFarePolluter │ POINT   │ 0.1%  │ always       │ fare_amount            │
  │ OutlierFarePolluter  │ POINT   │ 1.3%  │ always       │ fare_amount            │
  │ DuplicatePolluter    │ POINT   │ 1.0%  │ always       │ all                   │
  │ OutOfOrderPolluter   │ POINT   │ 0.5%  │ temporal     │ pickup_datetime        │
  └──────────────────────┴─────────┴───────┴──────────────┴────────────────────────┘
  Point anomaly: tổng = 5.0% (valid/test), dùng để benchmark model detect point anomalies.

  ════════════════════════════════════════════════════════
  CONCEPT DRIFT POLLUTERS (chi áp dụng cho PRODUCTION)
  ════════════════════════════════════════════════════════
  3 loại concept drift KHÔNG phải anomaly point — MemStream ADAPT
  chứ không detect dưới dạng anomaly. Production = 5% point + 3% drift.

  ┌──────────────────────────────────────────────────────────────────────────────┐
  │ 1. AbruptDriftPolluter (CONCEPT — Sudden Shift)                              │
  ├──────────────────────────────────────────────────────────────────────────────┤
  │ Trigger    : time > drift_point (dp) — chọn ngẫu nhiên 1 ngày trong period  │
  │ Mechanic   : fare_amount = fare_amount × 1.5 (tăng đột ngột 50%)            │
  │             trip_distance giữ nguyên → fare_per_mile tăng tương ứng            │
  │ Scope      : tất cả records SAU dp trong production (30% records sau dp)     │
  │ Validation : before/after mean(fare_amount) nên có gap rõ ràng ở dp         │
  │ Ground Truth: dp_timestamp → dp_timestamp + end_of_production                │
  └──────────────────────────────────────────────────────────────────────────────┘

  ┌──────────────────────────────────────────────────────────────────────────────┐
  │ 2. IncrementalDriftPolluter (CONCEPT — Gradual Covariate Shift)               │
  ├──────────────────────────────────────────────────────────────────────────────┤
  │ Trigger    : always — bắt đầu từ ngày đầu của production                   │
  │ Mechanic   : fare_amount = fare_amount × (1 + 0.3 × progress)                │
  │             progress = (current_idx - start_idx) / total_records             │
  │             → tăng tuyến tính 0% → 30% trong toàn bộ production period      │
  │ Scope      : tất cả records trong production period                         │
  │ Validation : rolling_mean(fare_amount) nên tăng dần, không jump             │
  │ Ground Truth: start_timestamp → end_of_production                            │
  └──────────────────────────────────────────────────────────────────────────────┘

  ┌──────────────────────────────────────────────────────────────────────────────┐
  │ 3. CyclicalDriftPolluter (CONCEPT — Periodic Seasonal Shift)                 │
  ├──────────────────────────────────────────────────────────────────────────────┤
  │ Trigger    : always — bắt đầu từ ngày đầu của production                   │
  │ Mechanic   : fare_amount = fare_amount × (1 + 0.15 × sin(2π × week/4))      │
  │             → chu kỳ 4 tuần: tuần 0 → peak (+15%), tuần 2 → trough (-15%)  │
  │             → mô phỏng seasonal demand cycle (weekend surge)                │
  │ Scope      : tất cả records trong production period                         │
  │ Validation : ACF(168h) nên có peak rõ ở lag 672 (4 tuần)                    │
  │ Ground Truth: start_timestamp → end_of_production (persistent)              │
  └──────────────────────────────────────────────────────────────────────────────┘

  LƯU Ý QUAN TRỌNG:
  - 6 loại POINT ANOMALY → MemStream CÓ THỂ detect (score cao bất thường)
  - 3 loại CONCEPT DRIFT → MemStream ADAPT (score thay đổi từ từ, không spike)
  - Tách ground truth mask riêng: point_mask vs drift_mask
  - Khuyến nghị: tạo ground truth events từ các tác động

      - (D) + per-neighborhood memory partitioning (10 neighborhoods)

  (F) AE + Memory + FE + Norm + NB + ContextBeta:
      - (E) + 80-cell ContextBeta (10 NB × 8 time contexts)
      - beta threshold varies per cell (fitted p95 per cell)
      - Memory update gated by ContextBeta

  (G) Full (Flow của ta):
      - (F) + ADWIN drift detector (restart/reconfigure on drift)
      - + Canary Rules (early warning before drift)

6. GAP ANALYSIS — NORMALIZATION DENOMINATORS

  VẤN ĐỀ:
  File src/features/memstream_vectorizer.py sử dụng hardcoded denominators:
    - _FARE_PER_MILE_DENOM: float = 2.5
    - _FARE_PER_MIN_DENOM: float = 0.67
    - _SPEED_DENOM: float = 12.0

  Các giá trị này là calibration từ benchmark v10, KHÔNG phải từ
  fitted statistics trên actual NYC taxi data. Nếu data distribution
  thay đổi (ví dụ: surge pricing thay đổi, TLC fare policy thay đổi),
  denominators này có thể không còn optimal và làm giảm model performance.

  ĐỘNG CƠ:
  - Hardcoded denominators tạo ra artificial feature engineering
  - Không đảm bảo generalizability khi data distribution shift
  - So với original MemStream paper (dùng min-max từ benchmark datasets),
    cách tiếp cận này có cùng hạn chế nhưng không transparent

  GIẢI PHÁP:
  1. Fit denominators từ CLEAN data statistics:
     - FARE_PER_MILE_DENOM = 95th percentile của fare_per_mile
     - FARE_PER_MIN_DENOM = 95th percentile của fare_per_min
     - SPEED_DENOM = 95th percentile của speed
  2. Ghi lại fitted values vào checkpoint metadata
  3. Validate bằng Ablation #7 (fitted vs hardcoded)

  CẬP NHẬT CODE:
  - scripts/compute_normalization_stats.py: Fit denominators trên clean data
  - src/features/memstream_vectorizer.py: Đọc fitted values từ config
  - results/ablation/ab_table_5.csv: So sánh hardcoded vs fitted

  VÀI TRÒ TRONG EVALUATION:
  - Ablation #7 cho phép quantify impact của normalization approach
  - Nếu fitted tốt hơn → production nên dùng fitted
  - Nếu không khác biệt → hardcoded vẫn acceptable


7. MEMSTREAM PAPER — CÁC LỖI CẦN TRÁNH

  Đọc kỹ arxiv 2106.03837 để tránh các lỗi cơ bản sau:

  1. MEMORY SIZE vs BUFFER SIZE:
     Paper: Memory lưu encoded representations, KHÔNG phải raw features.
     Current code: MemoryModule lưu AE output (out_dim=34) — ĐÚNG.

  2. NORMALIZATION:
     Paper: Dùng z-score trên AE output trước khi tính L1 distance.
     Current code: Features được normalized rồi passed qua AE.
     Review xem output AE đã được z-score normalized trước khi kNN chưa.

  3. AE TRAINING:
     Paper: Train AE trên NORMAL samples only.
     Current code: warmup() filter anomalies — CẦN XÁC NHẬN lại.
     Review xem warmup có đang filter đúng hay không.

  4. kNN DISTANCE:
     Paper: Dùng L1 (Manhattan) distance.
     Current code: Dùng L1 — ĐÚNG.

  5. GAMMA DECAY:
     Paper: Gamma decay cho neighbors (closer = higher weight).
     Current code: gamma=0.0 skip decay, gamma>0 dùng exponential decay.
     Review xem gamma decay implementation đúng với paper chưa.

  6. THRESHOLD FITTING:
     Paper: Beta = percentile(train_scores, 100 - contamination_rate).
     Current code: ContextBeta dùng 95th percentile (5% contamination).
